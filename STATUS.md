## Probe 1 ŌĆö supported named groups (COMPLETE)

Via `SSLContext.getDefault().getSupportedSSLParameters().getNamedGroups()`
(`src/ListGroups.java`), non-null on both JDKs:

| | JDK 26.0.1 (GA) | JDK 27 EA+30 |
|---|---|---|
| Supported | x25519, secp256r1, secp384r1, secp521r1, x448, ffdhe2048, ffdhe3072, ffdhe4096, ffdhe6144, ffdhe8192 (10) | **X25519MLKEM768**, x25519, secp256r1, secp384r1, secp521r1, x448, ffdhe2048, ffdhe3072, ffdhe4096 (9) |
| Default | same as supported | same as supported |

- X25519MLKEM768 present on 27 EA (JEP 527 confirmed), absent on 26. It is
  FIRST in the default preference order on 27.
- MLKEM768 (pure) absent on both ŌĆö out of JEP 527 scope, as expected.
- Incidental: ffdhe6144/ffdhe8192 no longer listed on 27 EA.
- Control (both JDKs): `setEnabledProtocols({"TLSv9.9"})` throws
  `IllegalArgumentException: Unsupported protocol: TLSv9.9`;
  `setNamedGroups({"NoSuchGroup12345"})` is accepted without exception.
- Spec note (src.zip, identical text in 26 and 27): the `setNamedGroups` javadoc
  says *"Providers should ignore unknown named group scheme names while
  establishing the SSL/TLS/DTLS connections"* and reserves
  IllegalArgumentException for duplicate/null/blank elements only. The silent
  drop is therefore documented API behavior, not an implementation deviation.

## Probe 2 ŌĆö silent-drop matrix (COMPLETE)

`scripts/run_matrix.sh`; BCJSSE server offers exactly ONE group per case via
`-Djdk.tls.namedGroups` (pinning verified: C1 forced x25519 although the JDK 27
client preferred hybrid; C2 negotiated X25519MLKEM768). Cells:
error surfaced? / handshake ok? / group actually negotiated (from ServerHello
key_share in the client debug trace).

| Case (client requests ŌåÆ server offers) | JDK 26.0.1 | JDK 27 EA+30 |
|---|---|---|
| C1 {X25519MLKEM768, x25519} ŌåÆ x25519 | no error / OK / **x25519** ŌĆö hybrid silently dropped (`Ignore unspecified named group: X25519MLKEM768`) | no error / OK / x25519 (hybrid was genuinely offered; server chose x25519 ŌĆö correct negotiation, not a drop) |
| C2 {X25519MLKEM768} ŌåÆ X25519MLKEM768 | SSLHandshakeException (handshake_failure) / failed / ŌĆö (client sent NO supported_groups at all) | no error / OK / **X25519MLKEM768** (JEP 527 works) |
| C3 {MLKEM768, x25519} ŌåÆ x25519 **KEY CASE** | no error / OK / **x25519** ŌĆö silently dropped (`Ignore unspecified named group: MLKEM768`) | no error / OK / **x25519** ŌĆö **STILL silently dropped** (`Ignore inactive or disabled named group: MLKEM768`) |
| C4 {MLKEM768} ŌåÆ MLKEM768 | SSLHandshakeException (handshake_failure) / failed / ŌĆö | SSLHandshakeException (handshake_failure) / failed / ŌĆö |
| C5 {NoSuchGroup12345, x25519} ŌåÆ x25519 (control) | no error / OK / **x25519** ŌĆö silently dropped | no error / OK / **x25519** ŌĆö **silently dropped** (`Ignore unspecified named group: NoSuchGroup12345`) |

Findings:
- **C3: the defect persists on JDK 27 EA post-JEP-527.** An application
  requesting {MLKEM768, x25519} gets classical x25519 with no exception and no
  warning. JDK 27 even *recognizes* the group (message says "inactive or
  disabled", from `SupportedGroupsExtension.java:200`, vs "unspecified" for
  unknown names) and still drops it silently.
- **C5: behavior is generic, not PQC-specific.** A nonsense group name is
  silently ignored on both JDKs; the connection succeeds classically.
- C4 nuance: when ALL requested groups are dropped, the client still sends a
  ClientHello ŌĆö with no supported_groups/key_share extensions at all
  (`Ignore key_share extension, no supported groups`, a WARNING visible only in
  the debug trace). Failure then surfaces only because the peer rejects the
  handshake; the alert type depends on the peer (BCJSSE 1.84 here:
  handshake_failure(40); the stock-JSSE server in the original Study B produced
  missing_extension). The client never fails locally.

## Probe 3 ŌĆö application detectability (COMPLETE)

After a successful handshake there is NO public API returning the negotiated
named group, on either JDK:
- Enumerated all public methods of SSLSession, ExtendedSSLSession,
  SSLParameters, SSLSocket, SSLEngine mentioning group/kex/KEM: only
  `SSLParameters.get/setNamedGroups` exist ŌĆö and `getNamedGroups()` after the
  handshake returns what was SET (verified: returns `MLKEM768,x25519` in C3
  even though x25519 was negotiated).
- `SSLSession.getValueNames()` is empty; getProtocol()/getCipherSuite() do not
  disclose the group (TLS 1.3 cipher suites are key-exchange-agnostic).
- JDK 27 EA adds nothing new here.
The only observation channels are `-Djavax.net.debug` text parsing or
reflection into `sun.security.ssl` internals.
